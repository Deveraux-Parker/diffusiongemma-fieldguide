01
The single most useful thing we found

Never ask it for a one-word answer.

Ask “reply with only the number” and the denoiser often stops at nothing — it returns an empty string. Ask the exact same question as a short sentence and it answers correctly almost every time. This one quirk was masquerading as “bad at counting.”

“Reply with ONLY the number”7 / 8 came back empty
empty
“Answer in one short sentence”0 / 8 empty · 8/8 correct
“List each line, then state the count”1 / 8 empty · 7/8 correct
Do this: ask for a short sentence and parse the value out yourself. The model can count, compute a maximum, and find a threshold-crossing perfectly — it just won’t hand you a bare token.
02
Where to put the information that matters

Two safe zones, one dead zone in between.

The model uses sliding-window attention (window ≈ 1024 tokens) with full attention only every sixth layer. The result: it reliably reads facts right before the question (recency) and facts near the very start (primacy) — but a fact stranded in the late-middle, roughly 600–1500 tokens before the question, gets lost.

It doesn’t miss — it makes things up

We asked for individual fields of a record stranded in the dead zone. It never said “not found” — it confabulated a plausible value from its priors, every time:

asked its temp (really 87.3°C)  →  “21.90°C” a normal reading
asked its code colour  →  “green” not even in the set
asked the exact code  →  “90002” a neighbour’s timestamp

A weak “something critical is here” gist survives ~7/10 of the time — but no specific field can be read, even a 1-of-6 colour. The same record in a safe zone: 10/10.

Watch out: dead-zone errors are fluent, confident, and wrong — the most dangerous kind. Never trust a value pulled from the middle of a long prompt without a checksum or source-quote to verify it.

04
Quirks worth knowing

How a diffusion model behaves differently.

These aren’t bugs so much as consequences of denoising a whole canvas at once instead of predicting one token at a time.

temperature

The temperature knob is nearly inert

From 0.0 to 2.0 the spread of outputs barely moves — and even at 0.0 it isn’t deterministic.

determinism

Temp 0 ≠ repeatable

Five identical requests at temperature 0 gave five wordings of the same idea. Meaning is stable; exact text is not. Don’t build byte-exact caches on its output.

5 / 5 unique wordings
latency steps

Cost rises one canvas at a time

Output stays perfectly coherent past 256 tokens — but each new 256-token block adds a full denoising pass, so latency jumps in steps rather than smoothly.

char-drop under load

It can shave a character off a value

Echoing codes in isolation is flawless (20/20). But reading a value out of a full context, it sometimes drops a leading character — 93.7→3.7. A retrieval artifact, not quantization.

20/20 clean echo
honesty

Mostly admits when a fact is absent

Told to say NOT FOUND when there’s no match, it did so 7 of 8 times — and hallucinated a code the other time.

7/8 honest · 1/8 made-up
throughput

A single-user speed demon

Because it denoises the whole canvas in parallel, short answers return in ~106 ms and throughput rises with output length — the opposite of a normal model.

05
Beating the dead zone

You can’t shout louder. You have to move.

If a fact has to live in the middle of a long prompt, can you rescue it by repeating it? We buried a fact in the dead zone and tried six ways to recover it. Duplicating it in place — even three copies back-to-back — does nothing. Putting a single copy in a safe zone fixes it completely.

Why: the dead zone isn’t weak emphasis — it’s a region the model can’t bind values from, so it confabulates instead. Three stacked copies are all still unreadable. One copy at the top, or just before your question, lands where the model can actually read it.
The pattern that works: keep the full record wherever it naturally lives, and put a compact index — just the key and its value — in a safe zone (a header at the top or the line right before your question). For exact IDs, carry a checksum and reject any answer that doesn’t match. Don’t waste tokens repeating the full payload inside the dead band.
06
Putting it to work

Six rules that make it reliable.

✓
Do

Put must-have facts in the first half of the prompt, or in the last ~500 tokens right before your question.

✓
Do

Stuck with a mid-prompt fact? Echo one copy into a top header or just before the question — don’t repeat it in place.

✕
Avoid

Don’t bury a critical fact in the late-middle of a long prompt (~600–1500 tokens before the question).

✓
Do

Ask for answers as a short sentence and parse them — never “reply with only X.”

✕
Avoid

Don’t trust a bare value from mid-context, and don’t rely on byte-exact reproducibility.

✓
Do

Lean into its strengths: front-loaded time-series, structured JSON, long coherent single-canvas output.

✓
Do

For exact IDs, ask for the value twice or use checksummed codes to survive the char-drop.

Methodology

All figures measured locally against a running vLLM server (dg-awq, AWQ/compressed-tensors INT4, RTX 4090, max_model_len 8000, single-user). Each data point is 5–10 trials; every headline was confirmed by reading raw model outputs, not just aggregate scores. The dead-zone mitigation and binding-vs-coverage results come from a hardened harness suite (timestamped runs, per-trial raw logs, miss-type categorization, Wilson confidence intervals). Full report and reproducible harnesses live in DIFFBENCH/ and NEWBENCH/ alongside REPORT.md.
