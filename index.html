01
The single most useful thing we found

Never ask it for a one-word answer.

Ask “reply with only the number” and the denoiser often stops at nothing — it returns an empty string. Ask the exact same question as a short sentence and it answers correctly almost every time. This one quirk was masquerading as “bad at counting.”

“Reply with ONLY the number”7 / 8 came back empty
empty
“Answer in one short sentence”0 / 8 empty · 8/8 correct
“List each line, then state the count”1 / 8 empty · 7/8 correct
Do this: ask for a short sentence (or JSON) and parse the value yourself. The model can count, take a maximum, and find a threshold-crossing perfectly — it just won’t hand you a bare token.
02
Where it reads, and where it lies

Two zones you can trust — and a middle that makes things up.

The model uses sliding-window attention (window ≈ 1024 tokens) with full attention only every sixth layer. Two regions stay reliable: facts near the very start (primacy) and facts in the last few hundred tokens before the question (recency). A fact in between can be lost — and it doesn’t fail honestly. It confabulates.

We asked for fields of a record stranded in the middle

It never said “not found.” It filled each slot with a plausible value from its priors — every time:

its temperature (really 87.3°C)  →  “21.90°C” a normal reading
its code colour  →  “green” not even in the set
the exact code  →  “90002” a neighbour’s timestamp

A weak “something critical is here” gist survives ~7/10 of the time — but no specific field can be read, not even a 1-of-6 colour. The same record in a safe zone: 10/10.

Watch out: middle-of-prompt errors are fluent, confident, and wrong — the most dangerous kind. Never trust a value from the middle of a long prompt without a checksum or source-quote to verify it.
03
The plot twist

The dead zone isn’t a distance. It moves with length.

We assumed the dead zone sat a fixed distance before the question. It doesn’t. Pin a fact at exactly 1000 tokens from the question and change only the prompt’s total length — retrieval flips on and off: perfect, gone, perfect, gone.

What this means: the failure is tied to a fact’s absolute position in the prompt — recurring “bad bands” (roughly every ~1900 tokens), most likely from block-aligned sparse attention. The same fact, the same distance from your question, is readable at one length and lost at another. Needle content (we tested adding a checksum) makes no difference.
So stop trying to pick a safe distance. Only two positions hold at every length: the very start, and the last few hundred tokens before the question. (An earlier version of this page called it a fixed “600–1500 tokens” band — that was this effect measured at a single prompt length, and over-generalized. Corrected here.)

05
Quirks worth knowing

How a diffusion model behaves differently.

These aren’t bugs so much as consequences of denoising a whole canvas at once instead of predicting one token at a time.

temperature

The temperature knob is nearly inert

From 0.0 to 2.0 the spread of outputs barely moves — and even at 0.0 it isn’t deterministic.

determinism

Temp 0 ≠ repeatable

Five identical requests at temperature 0 gave five wordings of the same idea. Meaning is stable; exact text is not. Don’t build byte-exact caches on its output.

5 / 5 unique wordings
latency steps

Cost rises one canvas at a time

Output stays perfectly coherent past 256 tokens — but each new 256-token block adds a full denoising pass, so latency jumps in steps rather than smoothly.

char-drop under load

It can shave a character off a value

Echoing codes in isolation is flawless (20/20). But reading a value out of a full context it sometimes drops a leading character — 93.7→3.7. A retrieval artifact, not quantization.

20/20 clean echo
honesty

Mostly admits when a fact is absent

Told to say NOT FOUND when there’s no match, it did so 7 of 8 times — and hallucinated a code the other time.

7/8 honest · 1/8 made-up
throughput

A single-user speed demon

Because it denoises the whole canvas in parallel, short answers return in ~106 ms and throughput rises with output length — the opposite of a normal model.

06
The architecture that works

Store in the middle. Answer from the edges.

You can’t make the middle readable, so don’t fight it. We ablated nine layouts for a record buried in a dead-zone position. The pattern is sharp: only the strategies that put the actual answer values in a safe zone work.

Pointers don’t work. A safe-zone pointer (“the record is window W5624”) returns the right window — then fills the values with incident_code: "none". It can say which record matters but can’t route the model back into the dead zone to read it.

The layout that hit 8/8

# raw records live anywhere — treat the middle as storage RAW RECORDS ... window_id=W1234 ... incident_code=RI-12345 checksum=RS-99999 RAW RECORDS ... # a compact EXACT-VALUE index in a safe zone (top and/or tail) RETRIEVAL_INDEX: window_id=W1234; incident_code=RI-12345; checksum=RS-99999 Question: Return JSON from RETRIEVAL_INDEX.

Why this and not duplication

A compact exact-value index hits 8/8 at the top or just before the question — and it’s far cheaper than copying the whole payload (which also works, but wastes tokens).

Because the failure mode is confident confabulation, carry a checksum and reject any answer that doesn’t match. Treat the raw middle as storage; the top/tail index is the model’s readable retrieval surface.

07
When to reach for it

It’s a generation engine, not a reader.

Nearly everything we found collapses into one limitation: reading an exact fact out of the middle of a long prompt. Generation, coherence, structured output, reasoning, speed — all strengths. So let it generate freely, and keep what it must read in the safe zones.

08
Putting it to work

Six rules that make it reliable.

✓
Do

Put must-retrieve values at the very top, or in the last few hundred tokens before your question.

✕
Avoid

Don’t bury critical values anywhere in the middle — there’s no safe middle distance; it shifts with total length.

✓
Do

If a fact lives mid-prompt, add a compact exact-value index (id + value + checksum) in a safe zone.

✕
Avoid

Don’t use a pointer (“see record X”) — it finds the record but invents the values. Carry the values themselves.

✓
Do

Ask for a short sentence or JSON and parse it — never “reply with only X.”

✓
Do

Validate exact IDs with a checksum or source-quote — the failure mode is confident, fluent confabulation.

Methodology

All figures measured locally against a running vLLM server (dg-awq, AWQ/compressed-tensors INT4, RTX 4090, max_model_len 8000, single-user). Each data point is 6–10 trials; every headline was confirmed by reading raw model outputs, not just aggregate scores — which is how the dead-zone story got corrected twice as we tested further. Reproducible harnesses (timestamped runs, per-trial raw logs, miss-type categorization, Wilson confidence intervals) live in DIFFBENCH/ and NEWBENCH/ with the full REPORT.md.
