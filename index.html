I
The survey

Two safe shores, and a shoal.

The model reads reliably in two places: near the very start of the context (primacy) and in the last few hundred tokens before your question (recency). Between them lies a narrow shoal — roughly 600–1500 tokens before the question — where a fact is lost. This is the depth profile, read from the harbour outward.

II
It isn’t a tide — it’s a grid

The hazard is a fixed 1024-block lattice.

The “shoal” isn’t weather — it’s a deterministic position law. The context is a grid of 1024-token blocks. A fact is readable everywhere except one trap: when it lands in the even-numbered block immediately before the question’s block. Nothing drifts; what looked like a moving hazard was just the document length sliding a fact across fixed grid lines.

Add lorem padding at the top — a dead fact wakes updistance to the question held constant at ~1015 tok · 12 casts/point
Why this is good news: a deterministic grid is navigable. You can compute exactly where the trap is (it’s ~768–2048 tokens before your question, on an even block) and route around it — or, as above, nudge the whole document by padding. We confirmed the period is exactly 2048 = two × the model’s 1024 sliding-window.
III
Why it’s dangerous

On the shoal, it invents the depth.

It never reports “no bottom found.” Asked for a field of a record stranded on the shoal, it fills the slot with a plausible value from its priors — fluently, confidently, wrongly.

its temperature (really 87.3°C)  →  “21.90°C” a normal reading
its code colour  →  “green” not even in the set
the exact code  →  “90002” a neighbour’s timestamp

A faint “something is here” reading survives ~7/10 of the time — but no exact value can be read, not even a 1-of-6 colour. The same record in safe water: 10/10.

The danger: a wrong sounding looks exactly like a right one. Never trust a value drawn from the middle of a long context without a checksum to verify it.
IV
The route

Route your cargo around the shoal.

Because the trap sits at a known, computable place — the even-block ~768–2048 tokens before the question — you can simply not sail through it. Measured reliability by zone: last ~600 tokens → ~97%; ~600–2048 out → ~61% (the trap); beyond ~2048 → ~92%. Keep must-read values in the harbour or the deep water; ballast the trap.

The result: four records to retrieve from a long document. Stranded on the shoal → 10/40 codes (all-four on 0/10 runs). Relocated to safe water with ballast left on the shoal → 38/40 codes (all-four on 9/10 runs). Same length, same model.

The load plan

# bulk records — anywhere outside the shoal RECORD ... RI-12345 RECORD ... # BALLAST — reserve ~768-2048 tokens before the question lorem ipsum dolor sit amet ... (filler) # harbour: an exact-VALUE index, right by the question INDEX: id=W1234 code=RI-12345 checksum=RS-99 Question: answer from INDEX.

Two more rules for the harbour

Carry values, not pointers. A note saying “see record W5624” returns the right id but then fills the fields with "none" — it can’t sail back to read them. Put the actual values in the harbour.

Carry a checksum. Because a wrong value is fluent and confident, verify exact ids against a checksum and reject mismatches.

VI
Handling notes

The ship’s quirks.

Consequences of denoising a whole canvas at once instead of writing one token at a time.

the big one

Never ask for one word

“Reply with only the number” returns an empty string 7/8 of the time. Ask for a short sentence and it answers correctly. Parse the value yourself.

temperature

The knob is nearly inert

From 0.0 to 2.0 the spread of outputs barely moves — and even at 0.0 it isn’t deterministic. Don’t cache byte-exact.

latency steps

Cost rises one canvas at a time

Output stays coherent past 256 tokens, but each new 256-token block adds a full denoising pass — latency jumps in steps.

char-drop

It can shave a character

Echoing codes in isolation is flawless (20/20); reading one from a full context it sometimes drops a leading character — 93.7→3.7.

20/20 clean echo
honesty

Mostly admits a gap

Told to say NOT FOUND when nothing matches, it does — 7 of 8 times.

determinism

Temp 0 ≠ repeatable

Five identical calls gave five wordings of the same idea. Meaning stable, exact text not.

VII
Standing orders

Six rules for a clean voyage.

✓
Do

Keep must-read values near the start, or in the last few hundred tokens before the question.

✕
Avoid

Don’t place a critical fact ~600–1500 tokens before the question — that’s the shoal. Reserve it for filler.

✓
Do

For mid-document facts, add a compact exact-value index (id + value + checksum) in the harbour.

✕
Avoid

Don’t use a pointer (“see record X”) — it names the record and invents the values.

✓
Do

Ask for a short sentence or JSON and parse it — never “reply with only X.”

Ship’s log

Every sounding taken locally against a running vLLM server (dg-awq, AWQ/compressed-tensors INT4, RTX 4090, max_model_len 8000, single-user). 6–10 casts per point; every claim was confirmed by reading raw outputs, not aggregates — which is how the hazard got re-charted from a “drifting shoal” to a fixed block grid as we surveyed further. The block-parity law was nailed by a 608-trial position decomposition and verified by a 13-agent analysis that re-explained every prior result (89–100% fit) and refuted artifact hypotheses. Honest caveat: the effect is high-confidence and deterministic, but the exact architectural cause of the even/odd asymmetry (sliding-window × absolute KV/decode grid) is medium-confidence and still under investigation — a 1024-token sliding window alone is parity-free.
